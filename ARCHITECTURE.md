## 0. TL;DR

- Корпоративный сайт юрфирмы по банкротству физлиц + лид-формы (→ Bitrix24) +
  блог + калькулятор.
- **Стек:** **Astro 5 + адаптер `@astrojs/node`** (`mode: 'standalone'`) ·
  **Preact** (Islands) · **TypeScript 5.9** · **Tailwind CSS 3.4** ·
  **Feature-Sliced Design**. БД нет — контент из Markdown (блог) и TS/JSON

- **Интеграции:** Bitrix24 (вебхук — **мёртв**, 401), Google Analytics 4,
  Яндекс.Метрика, **reCAPTCHA v3 — используется** (серверная проверка, §6).
- **Состояние:** **в production с октября 2025, заморожен на 08.11.2025.** Прод

## 2. Принципы разработки

- **Не хардкодим — ничего.** Тексты/параметры/цены/URL/ключи — в контенте /
  конфигах / env. Репозиторий **публичный** → секреты только в env; `.env` не
  коммитим (есть `.env.example`).
- Тонкие компоненты; логика — в нужных слоях FSD (`features/`, `shared/lib/`).
  SoC / SOLID.
- **Границы FSD — жёсткое правило:** слой использует только нижележащие;
  features не знают друг о друге; shared независим. Полная таблица слоёв —
  `CLAUDE.md` (repo root).
- Тест-инфраструктуры сейчас нет (см. §1) — до появления реального раннера «не
  ломаем тест-инфраструктуру» не применимо буквально; при добавлении тестов
  заводить с нуля осознанно, не по шаблону из устаревших доков.
- **Security-чувствительное** (PII-формы, вебхук Bitrix24, GA4-хэши, ключи
  reCAPTCHA) — узкими отдельными промтами, не внутри больших арок.

  **пропускается** (совпадает с поведением прода). Порог не менять без явного
  решения владельца (BL-062).

  любым из `TESTING=true` / `STAGING=true` / `NODE_ENV=test`; каждое

## 7. Безопасность (заголовки и SAST)

### CSP: их два, и боевой — не тот, что в репозитории

Это контринтуитивно и раньше было задокументировано неверно («middleware ставит
заголовки на каждый ответ»). Фактически **три режима**:

| Что отдаётся                                            | Кто ставит заголовки                                                       |
| ------------------------------------------------------- | -------------------------------------------------------------------------- |
| HTML (`location /`, статика с диска)                    | **только nginx** — middleware сюда не доходит                              |
| `/api/`, `@ssr`                                         | **оба**: nginx (`always`) + ответ Node → браузер применяет **пересечение** |
| `/_astro/`, `/chunks/`, `/assets/`, файлы по расширению | **никто** — см. BL-070                                                     |

- **Почему middleware не защищает HTML.** При `output: 'static'` middleware
  отрабатывает для пререндеренных роутов **на этапе сборки**; на диск пишется
  только тело HTML, заголовки `Response` отбрасываются. On-demand роутов всего
  два (`api/form.ts`, `health.ts`), и оба отдают не-HTML, где CSP бесполезен.
  **Итог: CSP из `src/middleware.ts` в проде не защищает ни одной страницы.**
- **Реальный CSP — `infra/nginx/zerodolg.ru.conf:82`**, и он слабее:
  - **`'unsafe-eval'` безусловно в проде** — middleware специально гейтит его
    под `import.meta.env.DEV`, nginx отдаёт всегда. **Самая крупная разница.**
  - `img-src 'self' data: https: blob: …` — схема `https:` разрешает **любой**
    HTTPS-источник, делая перечисление хостов после него бессмысленным.
  - **нет `base-uri` и `form-action`** — у обоих **нет фолбэка на
    `default-src`**, то есть они **падают открытыми**. Для сайта, весь смысл
    которого — лид-формы, отсутствующий `form-action` (эксфильтрация через
    подставленный `target`) — реальная дыра, а не формальность.
  - нет `upgrade-insecure-requests` (частично компенсирован HSTS preload) и
    `object-src` (падает на `default-src 'self'` — мягко, не «без ограничений»).
  - `script-src` дополнительно пускает `cdn.jsdelivr.net`, `*.bitrix24.ru`,
    `apis.google.com`, `googleapis.com`, `recaptcha.net`, `yastatic.net`.
- **Обратно, nginx не строго слабее:** он **добавляет** HSTS preload и
  `X-XSS-Protection`, которых у middleware нет; но его
  `X-Frame-Options: SAMEORIGIN` слабее, чем `DENY` в middleware.
- **Практическое правило:** новый сторонний домен для скрипта — править **оба**
  allowlist'а, и nginx'овый — тот, что реально действует. Сведение к одному
  источнику правды — **BL-069**.

### SAST

- **CSP-nonce не существует.** `README.md:366` («Content Security Policy с nonce
  подходом») — **единственное вхождение слова `nonce` во всём репозитории**;
  генерации нет, `nonce-` в CSP нет. Оба CSP вместо этого держат
  `'unsafe-inline'` в `script-src` — прямая противоположность nonce-подходу.
- **Semgrep** (`npm run tools:semgrep`) — SAST. **Репо-скрипт
  (`tools/semgrep-scan.js`) не запускается** — `npx semgrep` резолвится в чужой
  npm-пакет-сквоттер (`semgrep@0.0.1`, без исполняемого файла), не в реальный
  semgrep (BL-002, тот же класс бага, что у TruffleHog ниже). Реальный скан
  (Python venv + PyPI `semgrep`, 14.07.2026, 280 community-правил/399 файлов)
  нашёл 4 находки уровня ERROR (Dockerfile без non-root `USER`,
  `shell:true`/`execSync` с переменной строкой в
  `scripts/deploy/deploy-complete.js` и `scripts/dev/setup-optimization.js`) +
  53 WARNING + 26 INFO — детали и приоритет в `docs/IMPLEMENTATION_PLAN.md`
  (BL-008/BL-009); не исправлено в этом housekeeping-промте, требует отдельного
  security-прохода.
- **TruffleHog** (`npm run tools:trufflehog`) — проверка утечки секретов;
  **репо-скрипт `tools/trufflehog-scan.js` даёт ложноположительный/недостоверный
  результат, когда сам инструмент недоступен через `npx`** — известный баг
  (BL-001), не доверять его выводу без перепроверки реальным TruffleHog.
  Реальный скан (Docker `trufflesecurity/trufflehog:latest`, все ветки/126
  коммитов, 14.07.2026) — **0 verified/unverified находок**, репозиторий чист на
  момент проверки. Критично: репо публичный, перепроверять реальным сканом
  периодически, не полагаться на репо-скрипт до фикса BL-001.
- Доступность: WCAG 2.2, ARIA, keyboard-навигация — не автоматизировано,
  проверять вручную.

## 10. Git и деплой

- **GitHub** (публичный, MIT). Дефолт `master`. Фичи → PR → `master`.
- **CI: локальный гейт — единственный.** GitHub Actions в проекте
  **неприменимы**: они не работают в РФ (биллинг/доступ). Настоящим CI всегда
  был локальный прогон, вызываемый скриптом деплоя. `.github/workflows/ci.yml`
  удалён 16.07.2026 как мёртвый артефакт (триггеры `main`/`develop` при дефолте
  `master`; шаг `npm run test` при отсутствующем скрипте; экшены на мутабельных
  тегах). **BL-003 закрыт как «не баг»** — чинить было нечего. Workflow не
  возвращать; зелёных чеков на GitHub не ждать, их не бывает.
- **Гейт** (обязателен перед STOP/мержем):
  `npm run type-check && npm run lint && npm run build && npm run build:prod`.
  `npm run test` в цепочку **не входит** — раннера нет (BL-032).
- ⚠️ На `master` включена ветка-защита «changes must be made through a pull
  request»; у владельца есть право обхода, и 16.07.2026 оно использовалось.
  Политика и практика расходятся — ‹решение владельца›.
- В `master` не попадают: `.env*` (кроме `.env.example`), `tmp/`,
  `docs.archive/`, `docs/_planner/`, память агента. Репо **публичный** → тем
  строже.
- История: без force-push / rebase / squash на опубликованных ветках.

### Деплой: пайплайна не существует

Не «уточнить у владельца» — **его нет**, и это установленный факт, а не пробел в
документации:

- На сервере нет `.git`, нет CI-агентов, webhook-listener'ов, git-хуков и
  деплой-скриптов; `/opt`, `/srv`, `/home` пусты, Docker не установлен. В
  истории команд — ни одного `rsync`/`scp`/`git pull`/`npm run build`. Cron —
  только уборка.
- **Что было:** цепочка на ноутбуке владельца — pre-build → build → post-build →
  `dist` → архив → бэкап на проде → раскатка → проверка доступности. **Ноутбук
  утерян вместе с дев-деревом.**
- **Что доехало в репозиторий:** только локальная половина.
  `scripts/deploy/deploy-complete.js` дёргает checklist/backup/build/verify —
  **транспорта нет** (grep по `rsync|scp|ssh|ansible` → 0). То есть
  `npm run deploy` не деплоит.
- Прод залит **вручную**, последний раз **08.11.2025**.
- **Это BL-063 и блокер WS-0.** Пока не сделано — выкатить нельзя ничего.

### Known unknown: мы делаем аппроксимацию прода, а не его восстановление

Наш SSR-конфиг **не равен** утерянному прод-конфигу. Доказательство не
умозрительное: при добавлении адаптера всплыла **CSS-регрессия** — кастомные
Rollup-шаблоны имён против реконсиляции ассетов, из-за чего CSS оседал в
`dist/server` (`000a61e`). Прод такого не показывает, значит собирался иначе. Мы
воспроизводим поведение прода **приблизительно**, по скомпилированному бандлу,
без исходников. Любое утверждение «у нас как на проде» требует проверки, а не
веры.
