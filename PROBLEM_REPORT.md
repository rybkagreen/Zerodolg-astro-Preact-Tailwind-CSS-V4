# Отчет о проблеме с запуском SSR-приложения в staging-окружении

## Общее описание проблемы

При попытке запустить SSR-приложение (Astro с @astrojs/node адаптером) в staging-окружении через Docker с использованием PM2 возникают проблемы с доступом к приложению на порту 3000. Приложение не отвечает на HTTP-запросы, хотя контейнер запускается и PM2 показывает, что приложение работает.

## Технические детали

### Архитектура приложения

- Astro v5.13.7 с SSR адаптером @astrojs/node
- Приложение собирается с использованием production конфигурации (astro.config.prod.mjs)
- Запуск планируется через PM2 в Docker-контейнере
- Приложение должно слушать на 0.0.0.0:4321 и быть доступно через порт 3000 (порт пробрасывается в Docker)

### Используемые файлы конфигурации

- `Dockerfile.staging.ssr` - Docker-контейнер для staging-окружения
- `ecosystem.config.staging.cjs` - конфигурация PM2 для staging
- `docker-compose.staging.ssr.yml` - docker-compose для запуска staging-окружения

## Попытки решения, которые не сработали

1. **Первый подход**: PM2 конфигурировался для запуска файла конфигурации как скрипта
   - Результат: PM2 создавал приложение с именем "ecosystem.config.staging" и запускал конфиг-файл как JavaScript-скрипт

2. **Второй подход**: Использование `pm2-runtime ecosystem.config.staging.cjs`
   - Результат: Та же проблема - файл конфигурации выполнялся как скрипт, а не как конфигурация

3. **Третий подход**: Запуск напрямую через `pm2-runtime start ./dist/server/entry.mjs`
   - Результат: Приложение запускается, но по-прежнему не принимает HTTP-запросы

## Симптомы проблемы

1. Приложение запускается в контейнере (PM2 показывает статус "online")
2. При проверке через `docker exec` видно, что Node.js процесс запущен
3. При попытке выполнить `curl -I http://localhost:3000` получаем ошибку "Connection refused"
4. Логи в `/app/logs/pm2-out.log` могут содержать старые записи или быть пустыми
5. Процесс в контейнере может запускаться на 127.0.0.1:4321 вместо 0.0.0.0:4321

## Возможные причины проблемы

1. **Проблема с настройками HOST**: Приложение может не использовать переменные окружения HOST/PORT должным образом
2. **Проблема с PM2**: PM2 может не устанавливать переменные окружения должным образом для приложений Astro
3. **Проблема с Astro SSR**: Построенный серверный entrypoint (`dist/server/entry.mjs`) может не учитывать переменные окружения при запуске
4. **Проблема с Docker**: Некоторые процессы могут не выходить за границы контейнера должным образом

## Текущий статус

## ✅ РЕШЕНИЕ (2025-10-06)

### Найденные проблемы

1. **PM2 wait_ready конфигурация**: PM2 был настроен с опцией `wait_ready: true`, но Astro SSR приложение не отправляет сигнал готовности, из-за чего PM2 ждал до таймаута, прежде чем считать приложение запущенным.

2. **Неправильный режим запуска PM2**: Скрипт `start-staging.sh` использовал `pm2 startOrRestart` с последующим `pm2 logs`, вместо `pm2-runtime` который правильно работает как PID 1 в Docker контейнерах.

3. **Отсутствие health endpoint**: Healthcheck в docker-compose проверял `/health`, но такого эндпоинта не существовало.

### Внесенные исправления

1. **ecosystem.config.staging.cjs**: Удалены опции `wait_ready: true` и `listen_timeout: 10000`, которые мешали корректному запуску приложения.

2. **start-staging.sh**: Изменен с `pm2 startOrRestart ecosystem.config.staging.cjs` на прямой запуск с параметрами:

   ```bash
   exec pm2-runtime /app/dist/server/entry.mjs \
     --name "zerodolg-backend" \
     --instances 1 \
     --max-memory-restart 1G \
     --error /app/logs/pm2-error.log \
     --output /app/logs/pm2-out.log \
     --time \
     --merge-logs
   ```

   Это решает проблему, когда PM2 воспринимал конфигурационный файл как скрипт для выполнения.
   Преимущества этого подхода:
   - PM2 правильно работает как PID 1 в контейнере
   - Корректная обработка сигналов (SIGTERM, SIGINT)
   - Автоматическое завершение контейнера при падении приложения
   - Все настройки PM2 в одном месте (в скрипте запуска)

3. **src/pages/health.ts**: Создан новый health check endpoint, который возвращает JSON с информацией о состоянии сервиса.

### Как проверить исправления

```bash
# 1. Пересобрать и запустить staging окружение
npm run staging:ssr:down
npm run staging:ssr:up

# 2. Проверить доступность приложения
curl -I http://localhost:3000
# Ожидаемый результат: HTTP/1.1 200 OK

# 3. Проверить health endpoint
curl http://localhost:3000/health
# Ожидаемый результат: {"status":"ok","timestamp":"...","service":"zerodolg-astro"}

# 4. Проверить логи PM2 и статус приложения
docker exec zerodolg-staging-ssr pm2 list
# Должно показать: name="zerodolg-backend", status="online"

docker logs zerodolg-staging-ssr
# Должно быть: "Server listening on http://localhost:4321"

# 5. Проверить что приложение слушает на правильном хосте
docker exec zerodolg-staging-ssr netstat -tlnp | grep 4321
# Ожидаемый результат: tcp 0 0 0.0.0.0:4321 0.0.0.0:* LISTEN 1/node

# 6. Проверить healthcheck Docker
docker ps | grep zerodolg-staging-ssr
# Должно быть: (healthy)
```

### Почему это работает

Astro с @astrojs/node адаптером в standalone режиме автоматически:

- Читает переменные окружения `HOST` и `PORT` (строки 1547-1548 в dist/server/chunks/_@astrojs-ssr-adapter.BydjtZi8.js)
- Запускает HTTP сервер на указанном хосте и порту
- Адаптер корректно обрабатывает `host: true` в конфигурации как "0.0.0.0"

Использование `pm2-runtime` вместо `pm2` в Docker:

- Позволяет PM2 работать как основной процесс (PID 1)
- Обеспечивает правильную передачу сигналов от Docker к приложению
- Автоматически выводит логи в stdout/stderr Docker
- Завершает контейнер при падении приложения
