# 📋 Текущая реализация системы оркестрации

## Обзор архитектуры

Система оркестрации для автоматической оптимизации проекта zerodolg-astro, использующая модели Qwen через CLI для анализа и модификации кода.

## Компоненты системы

### 1. Оркестраторы

#### orchestrator.py (базовый)
- **Класс**: `TaskOrchestrator`
- **Функции**:
  - Управление очередью задач в памяти (`deque`)
  - Сканирование активных worker'ов через файлы в `workers/`
  - Распределение задач только idle worker'ам
  - Сбор результатов из `results/`
  - Генерация новых задач на основе результатов
- **Проблемы**:
  - Не сохраняет состояние между сессиями
  - Задачи теряются при перезапуске
  - Не может назначать задачи worker'ам в статусе "working"

#### persistent_orchestrator.py
- **Класс**: `PersistentOrchestrator(TaskOrchestrator)`
- **Улучшения**:
  - Сохранение состояния в `state/` (JSON файлы)
  - Восстановление задач при перезапуске
  - Обнаружение зависших задач (stalled tasks)
  - Автоматическая очистка старых результатов
- **Файлы состояния**:
  - `orchestrator_state.json` - счетчики и метаданные
  - `task_queue.json` - очередь задач
  - `pending_tasks.json` - задачи в работе
  - `completed_tasks.json` - история выполнения

#### project_optimizer.py
- **Класс**: `ProjectOptimizer(TaskOrchestrator)`
- **Специализация**: Создание production-оптимизационных задач
- **Анализирует**:
  - Структуру проекта (файлы, папки)
  - package.json зависимости
  - astro.config.mjs конфигурацию
  - Исходные файлы и ассеты

### 2. Worker'ы

#### worker.py (базовый)
- **Класс**: `QwenWorker`
- **Функции**:
  - Регистрация в `workers/{id}.json`
  - Проверка задач в `tasks/task_{worker_id}.json`
  - Вызов qwen через subprocess
  - Сохранение результатов в `results/`
- **Проблемы**:
  - Нет доступа к файлам проекта
  - Не может редактировать файлы

#### enhanced_worker.py
- **Класс**: `EnhancedQwenWorker(QwenWorker)`
- **Улучшения**:
  - Доступ к корню проекта
  - Использование `--approval-mode auto_edit` для редактирования
  - Контекст проекта в промптах
  - Fallback анализ при сбое qwen
- **Проблемы**:
  - `--all-files` вызывал таймауты (исправлено)
  - Таймаут 60 сек был слишком большой (уменьшен до 30)

#### documentation_worker.py
- **Класс**: `DocumentationWorker(EnhancedQwenWorker)`
- **Специализация**: Документирование изменений
- **Функции**:
  - Ведение лога изменений в `docs/OPTIMIZATION_CHANGES.md`
  - Создание отчетов в `docs/reports/`
  - Специальные типы задач: document, update

### 3. Скрипты запуска

#### PowerShell скрипты
- `start_orchestrator.ps1` - запуск базового оркестратора
- `start_full_optimization.ps1` - запуск всей системы
- `restart_stalled_workers.ps1` - перезапуск зависших worker'ов
- `cleanup.ps1` - очистка системы

#### Python скрипты
- `start_production_optimization.py` - создание production задач

## Workflow

1. **Инициализация**:
   - Оркестратор создает начальные задачи
   - Worker'ы регистрируются в системе

2. **Распределение задач**:
   - Оркестратор сканирует idle worker'ов
   - Создает файл `task_{worker_id}.json`
   - Worker забирает файл и удаляет его

3. **Выполнение**:
   - Worker вызывает qwen с промптом
   - Обновляет heartbeat в своем файле регистрации
   - Сохраняет результат в `results/`

4. **Сбор результатов**:
   - Оркестратор читает файлы из `results/`
   - Перемещает их в `archive/`
   - Генерирует новые задачи на основе результатов

## Выявленные проблемы

### 1. Проблема с зависанием worker'ов
- **Причина**: qwen зависал с флагом `--all-files` на большом проекте
- **Симптомы**: Worker'ы оставались в статусе "working" часами
- **Решение**: Убрать `--all-files`, уменьшить таймаут

### 2. Проблема с очередью задач
- **Причина**: Задачи назначаются только idle worker'ам
- **Симптомы**: Задачи накапливаются в очереди, но не создаются файлы
- **Решение**: Нужен механизм переназначения задач

### 3. Проблема с форматом дат
- **Причина**: Несоответствие форматов дат в PowerShell и Python
- **Симптомы**: Скрипты PowerShell не могут парсить даты
- **Решение**: Использовать ISO формат везде

### 4. Проблема с импортами
- **Причина**: Циклические зависимости между модулями
- **Симптомы**: persistent_orchestrator импортирует orchestrator
- **Решение**: Нужна более чистая архитектура

## Рекомендации для новой реализации

### Архитектурные улучшения
1. **Единый оркестратор** с встроенной персистентностью
2. **Базовый класс Worker** с расширяемой архитектурой
3. **Четкое разделение**: core, workers, tasks, utils

## Метрики производительности

- **Среднее время выполнения задачи**: 30-60 секунд
- **Проблема с таймаутами**: задачи зависали на 30+ минут
- **Количество файлов в проекте**: 20,000+
- **Размер очереди**: до 10+ задач
- **Параллельных worker'ов**: 2-3

## Выводы

Текущая реализация работоспособна, но имеет архитектурные проблемы:
1. Сложная иерархия наследования
2. Дублирование кода между модулями
3. Отсутствие единой точки конфигурации
4. Проблемы с масштабированием
5. Недостаточная обработка ошибок

Система требует рефакторинга с учетом выявленных проблем и современных практик.
