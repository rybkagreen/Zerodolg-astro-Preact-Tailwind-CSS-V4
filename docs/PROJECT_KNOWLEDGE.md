## История расхождения репо ↔ прод (зафиксировано 16.07.2026)

Понимать это обязательно перед любой оценкой работ:

- **Прод собран не из этого репозитория, и он новее его.** На сервере нет ни
  `.git`, ни исходников — только скомпилированные артефакты. Конфига, которым
  собран прод, в репозитории не было.
- **Дев-дерево утеряно вместе с ноутбуком владельца.** Исходников кода,
  наработанного между последним push и последним деплоем (08.11.2025), **не
  существует**. Единственный след — прод-бандл, снятый в `tmp/prod-snapshot/`
  (54 MB, gitignored).
- **Деплой был цепочкой на этом ноутбуке:** pre-build → build → post-build →
  `dist` → архив → бэкап на проде → раскатка → проверка доступности. В
  репозиторий доехала только локальная половина
  (`scripts/deploy/deploy-complete.js` дёргает checklist/backup/build/verify);
  транспорта нет — grep по `rsync|scp|ssh|ansible` даёт 0. Это BL-063.
- **Прод залит вручную**, одноразово; следов автоматизации на сервере нет (нет
  CI-агентов, webhook-listener'ов, git-хуков, деплой-скриптов).
- **Направление работы поэтому обратное ожидаемому:** не «выкатить наше», а
  сперва «вобрать то, что на проде» — иначе деплой станет **регрессом**
  (потеряли бы reCAPTCHA, логгер, testing-mode, `/health`). Эта работа выполнена
  в ветке `feat/restore-ssr` (BL-055/057/058/059/061 закрыты).
- **Аппроксимация, а не восстановление.** Наш SSR-конфиг **не равен** утерянному
  прод-конфигу: при добавлении адаптера всплыла CSS-регрессия (кастомные
  Rollup-шаблоны имён vs реконсиляция ассетов), то есть прод собирался иначе. Мы
  воспроизводим прод приблизительно — фиксируем честно.

## Key learnings & principles

**4 архитектурных принципа (hard constraint во всех промтах):**

1. Правильно сразу — не переделывать.
2. Архитектурно чисто (SOLID, SoC, без патчей поверх патчей); **не нарушать
   границы FSD-слоёв**.
3. Не хардкодим — тексты/параметры/цены/URL/ключи в config / content / env.
   Репозиторий **публичный** → секреты только в env, `.env` никогда не коммитим.
4. Видим лучше → делаем; quick-and-dirty отклоняем, defer как BL.

Накопленные уроки (этот проект):

- Сайт **в проде** — любое изменение с повышенной осторожностью, gate-checks
  обязательны.
- Формы — самый тяжёлый по интерактиву блок. **Метрика «~12 форм» из аудита
  ошибочна:** реально **14 инстансов уже на одной базе** (`BaseForm.astro` →
  `FormEnhancedFinal.tsx`). Работы «унифицировать 12 копипаст» не существует;
  реальная задача — 3 параллельные реализации + дубли валидации (22–32 ч), и она
  **заблокирована** решением по `/api/form`. Новые формы делать на базе, не
  плодить с нуля.
- **Не добавлять** фейковые триггеры (счётчик «онлайн», таймеры «Акция
  завершена»), keyword-stuffing — это активный минус для доверия и SEO (см.
  дизайн-бриф).
- PII в формах (имя/телефон/сумма долга), вебхук Bitrix24, GA4/Метрика, ключи
  reCAPTCHA — **security-чувствительно**, узкими отдельными промтами.
- Дизайн-система — на **Tailwind + токенах** (config), не на «магических»
  классах.

### Главный урок проекта: «доки ≠ реальность»

Это не разовая оплошность, а **устойчивый паттерн**. К 16.07.2026 подтвердился
**шесть раз**, каждый раз — документ уверенно утверждал то, чего нет:

1. **Тесты, которых нет.** README и CI описывали Vitest/Testing Library и
   покрытие; в `package.json` нет ни скрипта `test`, ни зависимости.
2. **CI, который никогда не запускался.** `.github/workflows/ci.yml` триггерился
   на `main`/`develop` при фактическом дефолте `master`. Вдобавок сама платформа
   в проекте неприменима (см. D3) — то есть чинить триггеры было бы работой
   впустую.
3. **Semgrep-сквоттер.** `npx semgrep` резолвился в чужой пустой пакет
   `semgrep@0.0.1`; репо-скрипт мог отрапортовать «чисто», ни разу не запустив
   сканер (BL-002; тот же дефект у TruffleHog — BL-001).
4. **«100% готовность»** — при сломанной статье блога и устаревшем sitemap.
5. **Прод, собранный не из этого репозитория** — и притом **новее** его (см.
   «История расхождения»). Самое дорогое расхождение: обесценило целый
   recon-отчёт по репо, писавшийся в обратной предпосылке.
6. **BL-006 (найдено 16.07.2026).** Утверждал: «13 уязвимостей, все в
   dev-цепочке `@lhci/cli`, в прод-бандл не попадают» — и на этом основании риск
   был принят. Факт: 14, и прод-путь затронут — `astro` несёт два **HIGH**
   (Reflected XSS via unescaped slot name; Host header SSRF) и MODERATE «XSS in
   `define:vars`». Последнее бьёт ровно в механизм утечки вебхука из BL-054.
   Основание для принятия риска было ложным.

**Правило:** проверять **фактом** — запуском, grep'ом, curl'ом, чтением конфига,
— а не чтением документации. Это относится и к настоящему файлу: он тоже уже
дважды ошибался. Формулировка «в доке написано X» не является доказательством X.

### Второй урок: зелёный гейт ≠ закоммичено

В PROMPT_02 правки были написаны и гейт был зелёным, но
`git add -A -- .qwen <другие пути>` **целиком аборнулся**, потому что pathspec
`.qwen` ни на что не матчился (каталог уже удалили отдельным `git rm`) — и ни
один из перечисленных далее путей не застейджился. Часть работы «прошла
проверку», но в коммит не попала; обнаружилось только следующим проходом
(`d19a520`).

**Правило:** после стейджа — `git status`, до коммита. Зелёные `type-check`/
`lint`/`build` ничего не говорят о содержимом индекса.

## Process rules

- **Git:** feature → `master` через PR (preserve history; без
  force-push/rebase/squash на опубликованных ветках). `master` = production.
  Секреты/локальная память агента/временное — не в репозиторий.
- **CI:** **только локальный гейт.** GitHub Actions в проекте **неприменимы**
  (не работают в РФ), `.github/workflows/ci.yml` удалён 16.07.2026 — см. D3.
  Обязателен локально: `npm run type-check && npm run lint && npm run build` (+
  `npm run build:prod` перед прод-сборкой). `npm run test` **не входит** —
  раннера нет (BL-032).
- **Политика значений (директива владельца, 16.07.2026):** в трекаемых файлах
  **только плейсхолдеры** для: вебхука Bitrix24, телефона, адреса. **Боевые
  значения допустимы** для: GA, GTM, Яндекс.Метрики, verification-токенов, site
  URL, email. ПДн лидов (имена/телефоны) — **никогда** в трекаемые файлы, только
  `tmp/` (gitignored) и лично владельцу. Репозиторий публичный.
- **Hook discipline:** stop-hook bound at 2 non-trivial acks (идентичные fires =
  1 ack) — если executor = Claude Code/Fable.
- **Fresh session per рабочий блок** (крупный поток/этап редизайна).
- **PROTECTED:** прод-`master`; живые формы и интеграция Bitrix24;
  SEO/аналитика/CSP (`CLAUDE.md` → «Working conventions» для точного списка
  путей).
