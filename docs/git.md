# Руководство по работе с Git

## Общие принципы

### Рабочий процесс
В проекте используется Git-flow подход с основными ветками:
- `main` — основная ветка, содержащая стабильный код продакшена
- `develop` — ветка разработки, в которую сливаются фичи
- `feature/*` — ветки для разработки новых функций
- `release/*` — ветки подготовки релизов
- `hotfix/*` — ветки срочных исправлений багов

### Именование веток
- `feature/` — для новых функций (например, `feature/user-authentication`)
- `bugfix/` — для исправления багов (например, `bugfix/login-error`)
- `hotfix/` — для срочных исправлений (например, `hotfix/security-patch`)
- `release/` — для подготовки релизов (например, `release/v1.2.0`)

## Коммиты

### Сообщения коммитов
В проекте используется формат conventional commits:

```
<type>(<scope>): <short summary>
<BLANK LINE>
<body>
<BLANK LINE>
<footer>
```

#### Типы коммитов:
- `feat`: Новая функция
- `fix`: Исправление бага
- `docs`: Изменения в документации
- `style`: Изменения, не влияющие на смысл кода (форматирование и т.п.)
- `refactor`: Изменения, не исправляющие баги и не добавляющие функции
- `test`: Добавление или изменение тестов
- `chore`: Другие изменения (сборка, вспомогательные скрипты и т.п.)

#### Примеры:
```
feat(auth): добавить аутентификацию пользователей

Добавлена система аутентификации с поддержкой JWT токенов
и OAuth через Google и GitHub.
```

```
fix(forms): исправить валидацию email полей

Починено регулярное выражение для проверки формата email
в компонентах форм.
```

### Частичные коммиты
Используйте `git add -p` для выборочной фиксации изменений, особенно когда вы внесли несколько изменений в один файл, но хотите зафиксировать их отдельно.

## Работа с ветками

### Создание новой ветки
```
git checkout develop
git pull origin develop
git checkout -b feature/your-feature-name
```

### Обновление своей ветки из develop
```
git checkout develop
git pull origin develop
git checkout your-branch-name
git merge develop
```

### Отправка ветки на удаленный репозиторий
```
git push -u origin your-branch-name
```

## Pull Requests

### Создание Pull Request
1. Убедитесь, что ваша ветка обновлена из `develop`
2. Убедитесь, что все тесты проходят
3. Опишите изменения в описании PR
4. Укажите, какие задачи решает PR (ссылки на GitHub Issues, если есть)
5. Запросите ревью у других разработчиков

### Заголовок Pull Request
Используйте тот же формат, что и для коммитов:
```
feat(auth): добавить аутентификацию пользователей
```

### Описание Pull Request
В описании укажите:
- Что было сделано
- Зачем это было сделано
- Как протестировать изменения
- Связанные задачи или баги (если есть)

## Проверка кода

### Code Review
Все изменения должны пройти проверку другими разработчиками перед слиянием.

### Требования к мержу
- Все тесты должны проходить
- Код должен соответствовать стилю проекта
- Должны быть проведены проверки линтером
- Требуется одобрение хотя бы одного другого разработчика

## Хуки Git

В проекте настроены хуки Git через Husky:
- `pre-commit`: Проверяет и форматирует файлы с помощью lint-staged
- `pre-push`: Выполняет проверки перед отправкой изменений

## Совместная работа

### Работа с чужими ветками
Если вы хотите внести изменения в чужую ветку:
```
git fetch origin
git checkout -b your-local-branch origin/remote-branch-name
```

### Решение конфликтов
1. Обновите свою ветку из `develop`
2. Попробуйте автоматическое слияние: `git merge develop`
3. Если есть конфликты, разрешите их вручную
4. Добавьте разрешенные файлы: `git add .`
5. Завершите мерж: `git commit`

## История коммитов

### Чистая история
- Используйте `git rebase` для сглаживания истории в своей ветке перед PR
- Избегайте мерджа "merge commits" в основные ветки
- Используйте `git rebase -i` для редактирования истории своей ветки

### Проверка изменений
- Используйте `git log --oneline -10` для просмотра последних 10 коммитов
- Используйте `git diff HEAD~3` для сравнения с коммитом 3 назад

## Полезные команды

### Проверка статуса
```
git status
```

### Проверка изменений
```
git diff
git diff --staged  # изменения, подготовленные к коммиту
```

### Отмена изменений
```
git checkout -- filename    # отменить изменения в файле
git reset HEAD filename     # удалить файл из индекса
git reset --soft HEAD~1     # отменить последний коммит, сохранив изменения
git reset --hard HEAD~1     # отменить последний коммит, потеряв изменения
```

### Работа с удаленным репозиторием
```
git fetch origin            # получить изменения с удаленного репозитория
git pull origin branch      # получить и слить изменения
git push origin branch      # отправить изменения
git push origin --delete branch  # удалить удаленную ветку
```
