# Руководство по стилю кода

## Форматирование

### Общие принципы
- **Согласованность**: Следовать существующим шаблонам в кодовой базе
- **Читаемость**: Код должен быть самодокументируемым
- **Простота**: Предпочитать простые решения сложным
- **Производительность**: Учитывать влияние на производительность

### Форматирование
- **Отступы**: 2 пробела (без табуляции)
- **Длина строки**: Максимум 100 символов для кода, 80 для комментариев
- **Точки с запятой**: Обязательны в JavaScript/TypeScript
- **Кавычки**: Одинарные кавычки для JavaScript, двойные для HTML атрибутов
- **Завершающие запятые**: Да для многострочных объектов/массивов

## Соглашения об именовании

### Файлы и папки
- **Компоненты**: PascalCase (например, `ReviewCard.astro`)
- **Секции**: PascalCase (например, `HeroSection.astro`)
- **Страницы**: kebab-case (например, `test-reviews.astro`)
- **Стили**: kebab-case с префиксом подчеркивания (например, `_reviews.css`)
- **Скрипты**: kebab-case (например, `reviews-carousel.js`)
- **Утилиты/Библиотеки**: kebab-case (например, `content-service.js`)

### Код
- **Компоненты**: PascalCase
- **Функции**: camelCase
- **Переменные**: camelCase
- **Константы**: UPPER_SNAKE_CASE
- **CSS классы**: Методология BEM (блок__элемент--модификатор)
- **CSS переменные**: kebab-case с префиксом (например, `--color-primary`)

## Стили CSS

### Методология BEM
```css
/* Блок */
.review-card { }

/* Элемент */
.review-card__header { }
.review-card__body { }

/* Модификатор */
.review-card--featured { }
.review-card__header--large { }
```

### Слои ITCSS
1. **Settings**: Переменные, конфигурации
2. **Generic**: Сброс, нормализация
3. **Elements**: HTML элементы
4. **Components**: UI компоненты
5. **Sections**: Секции страниц
6. **Utilities**: Вспомогательные классы

#### 2. Расположение компонента
- **Статические компоненты**: `/src/components/ui/` или `/src/components/sections/`
- **Интерактивные компоненты**: `/src/components/islands/`
- **Макеты**: `/src/layouts/`

#### 3. Чек-лист компонента
- [ ] Интерфейс TypeScript для props

### Изменение существующих компонентов

#### Чек-лист перед изменением
1. **Понимание текущей реализации**
   - Тщательное изучение существующего кода
   - Проверка CSS зависимостей
   - Поиск JavaScript взаимодействий
   - Анализ использования компонента на страницах

2. **Анализ влияния**
   - Список всех страниц, использующих компонент
   - Проверка наследования стилей
   - Проверка JavaScript зависимостей
   - Тестирование адаптивного поведения

3. **Процесс изменения**
   - Создание резервной копии или ветки
   - Внесение инкрементальных изменений
   - Тестирование после каждого изменения
   - Обновление документации
   - Тестирование на разных размерах экрана

## Критические требования

### Абсолютные запреты
- **НЕТ INLINE СТИЛЕЙ**: НИКОГДА не использовать `style="..."` или `style={...}` атрибуты
- **НЕТ !important**: НИКОГДА не использовать `!important` в CSS
- **НЕТ глобальных стилей**: Кроме как в специально отведенных файлах
- **НЕТ прямой манипуляции DOM**: Использовать Preact для интерактивных компонентов
- **НЕТ блокирующих скриптов**: Все скрипты должны быть async или deferred

### Обязательные практики
- **Только CSS классы**: Вся стилизация через CSS классы
- **Методология BEM**: Строгое соблюдение BEM для именования
- **Прогрессивное улучшение**: Должно работать без JavaScript
- **Семантический HTML**: Использование правильных HTML5 семантических элементов
- **ARIA метки**: Для всех интерактивных элементов
- **Alt текст**: Для всех изображений
- **Mobile First**: Дизайн для мобильных, улучшение для десктопа
