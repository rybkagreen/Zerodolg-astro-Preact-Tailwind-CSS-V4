## Правила версионирования

- **Major (X.0):** структурные изменения, новые разделы, смена методологии — **а
  также смена модели реальности**. 16.07.2026 — именно такой случай: прод
  оказался SSR, новее репозитория и без деплоя, поэтому
  PROJECT_KNOWLEDGE/ARCHITECTURE/IMPLEMENTATION_PLAN ушли в 2.0, а не в 1.2.
- **Minor (1.X):** дополнения, уточнения, новые записи (в т.ч. факт-проверка
  технических утверждений против репозитория).
- Значимое изменение → поднять версию здесь + дату.
- HANDOFF не версионируется (перезаписывается) — фиксируется только дата.
- **Опровергнутое утверждение не правится молча.** Прежняя формулировка
  называется явно («было → стало»), иначе следующая сессия восстановит её по
  памяти. Так сделано с D3, с «Astro static» и с BL-006.

## Проверка актуальности

В начале сессии Claude.ai сверяет: стек/версии / git policy / locked decisions →
пометить затронутое «Требует обновления» → обновить (сразу, если критично, или в
закрытие блока).

**Технические утверждения сверять с фактом — запуском, `grep`, `curl`, чтением
конфига, — а не с другими документами.** К 16.07.2026 расхождение «доки ≠
реальность» подтвердилось **шесть раз**: несуществующие тесты; CI, который не
запускался и не мог; semgrep-сквоттер; «100% готовность» при сломанной статье;
прод, собранный не из этого репозитория; ложное основание для принятия риска в
BL-006. Разбор — `docs/PROJECT_KNOWLEDGE.md` → «Key learnings».

**Отложенная работа фиксируется в таблицу Backlog**
(`docs/IMPLEMENTATION_PLAN.md` → «Backlog (BL)») со следующим свободным ID — не
в свободный текст и не в комментарий кода.
