# Чек-лист перед Pull Request

> Важно: серверный CI (GitHub Actions) временно недоступен, поэтому все проверки прогоняем локально перед пушем. Каждый PR обязательно проходит code review от владельца репозитория. Прямой push в `master` запрещён.

## Ветки

- Работаем от ветки `master`.
- Имя ветки: `feature/<описание>` для новых функций или `fix/<описание>` для исправлений.

## Перед пушем прогнать локально

```bash
npm run lint
npm run type-check
npm run test
npm run build
```

Все команды должны завершаться без ошибок. Husky прогоняет часть проверок на pre-commit, но не полагайтесь только на него — прогоняйте список выше вручную.

## Коммиты

Используем [Conventional Commits](https://www.conventionalcommits.org/):

- `feat` — новая функция
- `fix` — исправление бага
- `docs` — изменения в документации
- `style` — форматирование, без изменения логики
- `refactor` — рефакторинг без новых функций и багфиксов
- `test` — добавление или изменение тестов
- `chore` — служебные задачи (сборка, зависимости и т.п.)

## Процесс Pull Request

1. Запушить ветку в репозиторий.
2. Открыть Pull Request в `master`.
3. Дождаться review и approval от владельца репозитория.
4. Merge выполняет/подтверждает владелец после одобрения.

## Деплой

Деплой в production выполняет **только владелец** со своей машины командой `npm run deploy`, так как файлы `.env` / `.env.production` хранятся локально и не попадают в репозиторий.

## Секреты и переменные окружения

- **Никогда** не коммитить `.env`, `.env.production` или другие секреты.
- При добавлении новой переменной окружения — обновить `.env.example` (без реальных значений).
